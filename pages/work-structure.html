WCAG 2.2 更新摘要（vs WCAG 2.1）

WCAG 2.2 於 2023 年 10 月 5 日成為 W3C 正式推薦標準。相比 WCAG 2.1，新增 9 條成功準則、移除 1 條。AA 等級共新增 4 條。

⚠ 對 AIA 專案的影響：需額外關注 Focus 可見性、觸控目標大小、拖曳替代方案、無障礙認證等新要求。

SC 編號 等級 名稱 說明 Elementor 環境下的測試方法
4.1.1 移除 A Parsing 解析 原要求 HTML 無解析錯誤。因現代瀏覽器已自動修正 HTML 錯誤，此項不再有意義，WCAG 2.2 中已移除。 不需測試。Elementor 產出的 HTML 由平台控制。
2.4.11 NEW AA Focus Not Obscured (Minimum)
焦點不被遮擋（最低）		 當元件獲得鍵盤焦點時，該元件不能被其他內容（如 sticky header、cookie banner、浮動選單）完全遮擋。至少要部分可見。 Tab 遍歷所有互動元素，特別注意 Elementor 的 Sticky Header 設定。如有 Cookie 同意橫幅外掛，檢查是否遮擋焦點。
2.5.7 NEW AA Dragging Movements
拖曳動作		 所有使用拖曳操作的功能，必須提供不需拖曳的替代方式。除非拖曳是功能必要的。 檢查 Elementor 使用的 Carousel/Slider Widget 是否有前/後按鈕替代拖曳滑動。如無拖曳功能則自動通過。
2.5.8 NEW AA Target Size (Minimum)
目標大小（最低）		 所有指標輸入目標至少 24×24 CSS 像素，或與相鄰目標之間有足夠間距。例外：行內連結、使用者代理控制項。 用瀏覽器 DevTools 量測 Elementor 按鈕、Icon、Nav Link 的實際大小。在 Elementor 的 Advanced 設定中調整 padding 確保尺寸。
3.2.6 NEW A Consistent Help
一致的幫助		 若網站提供幫助機制（聯絡方式、FAQ、聊天機器人），這些機制在不同頁面必須出現在相對一致的位置。 在 Home 和 Events 頁面之間比對幫助功能的位置是否一致。Elementor 使用 Template 可確保 Header/Footer 一致。
3.3.7 NEW A Redundant Entry
重複輸入		 使用者在同一流程中先前已輸入的資料，要嘛自動填入、要嘛提供選擇。不要求使用者重複輸入相同資訊。 走完所有 Elementor Form Widget 的流程，檢查是否有要求重複輸入。Phase 1 範圍內表單可能較少。
3.3.8 NEW AA Accessible Authentication (Minimum)
無障礙認證（最低）		 認證流程不能要求認知功能測試（如記憶密碼、解謎），除非提供替代方式。 Phase 1 如無會員登入功能，此項不適用。如有，確認支援密碼管理器自動填入、無 CAPTCHA 或有替代方式。
2.4.12 NEW AAA Focus Not Obscured (Enhanced)
焦點不被遮擋（增強）		 獲得焦點的元件必須完全可見，不能有任何部分被遮擋。（2.4.11 的增強版） 同 2.4.11，但標準更嚴格。
2.4.13 NEW AAA Focus Appearance
焦點外觀		 焦點指示器必須有足夠的面積和對比度。 Elementor 對 focus 樣式的控制有限。需檢查主題 CSS 是否提供足夠的 focus indicator。
3.3.9 NEW AAA Accessible Authentication (Enhanced)
無障礙認證（增強）		 認證流程不能要求認知功能測試，且不能僅依賴辨認物件/圖片作為替代。 同 3.3.8，但排除 object/image recognition 作為替代方式。

對 AIA 專案的具體影響

必須滿足（AA 等級，共 4 條新增）：

  • 2.4.11 Focus Not Obscured — 如果用 Elementor 設了 sticky header 或安裝了 cookie 同意橫幅，確保不會遮住鍵盤焦點
  • 2.5.7 Dragging Movements — 若用 Carousel/Slider Widget，需確保有上一張/下一張按鈕
  • 2.5.8 Target Size — Elementor 按鈕和連結的 padding 要足夠，確保至少 24×24px
  • 3.3.8 Accessible Authentication — Phase 1 可能不適用（看是否有會員功能）

建議也達成（A 等級，共 2 條新增）：

  • 3.2.6 Consistent Help — 用 Elementor Template 確保 Header/Footer 一致即可
  • 3.3.7 Redundant Entry — Elementor Form Widget 不重複要求相同資訊

可選擇挑戰（AAA 等級）：

  • 2.4.12, 2.4.13, 3.3.9 — 可作為額外加分項目挑戰

01 Page Title 頁面標題
▶

💡 白話說明

看瀏覽器最上面的標籤，上面顯示的文字就是「頁面標題」。視障者打開一個網頁，螢幕閱讀器第一個唸出來的就是這個標題。所以每個頁面的標題都要不一樣，而且要能讓人一聽就知道這是哪個頁面。

📋 技術規範

SC 2.4.2 Page Titled (A) — 每個頁面必須有描述性的標題，能讓使用者了解頁面的目的和內容。

🔍 測試步驟

  • 在 WordPress 後台確認每個頁面有填寫標題
  • 如果安裝 SEO 外掛（如 Yoast），在 SEO Title 欄位設定更精確的標題
  • 確認 title 具有描述性，能區分不同頁面（不是所有頁面都叫「AIA」）
  • 建議格式：「頁面名稱 — APAC AIA」或「APAC AIA | 頁面名稱」
  • 螢幕閱讀器測試：開啟頁面時是否朗讀出有意義的標題
頁面標題在 WordPress 的「頁面 → 編輯」中設定，不是在 Elementor 編輯器裡。Elementor 的 Site Settings 可設定全站標題格式。

02 Informative Images 資訊性圖片 🔴 跨專案經驗 🟡 常見高風險
▶

💡 白話說明

網頁上有些圖片是有意義的（例如贊助商 Logo、活動照片），視障者看不到圖片，需要靠「替代文字」來知道圖片在表達什麼。如果圖片有意義卻沒有替代文字，就像有人指著一個東西說「你看這個」但你根本看不到。

📋 技術規範

SC 1.1.1 Non-text Content (A) — 所有傳達資訊的圖片必須有等效的替代文字。

🔴 跨專案經驗

在另一個無障礙評測專案中，Footer 的贊助商 Logo 被設成「空白替代文字」，結果螢幕閱讀器完全跳過它——但它其實在傳達「這是贊助商」的重要資訊。而且因為 footer 出現在每一頁，等於每一頁都扣分。

🔍 測試步驟

  • 在 Elementor Image Widget 的設定中檢查每張圖片是否有填寫 Alt Text
  • 資訊性圖片：alt 必須描述圖片傳達的資訊（如「KOOP 贊助商標誌」）
  • 裝飾性圖片：alt 留空（Elementor 的 Image Widget 可留空 alt 欄位）
  • 確認沒有使用檔名作為 alt（如「IMG_0234.jpg」）
  • WordPress 媒體庫：上傳圖片時就填好 alt text（會自動帶入 Elementor）
  • 用螢幕閱讀器聽聽看每張圖片唸出什麼
Ally 外掛的 AI 功能可以建議 alt text，但最終仍需人工審核。Ally Assistant 可以掃描全站缺少 alt text 的圖片。
Image Widget → Content → Image → Alt Text 欄位。或在 WordPress 媒體庫 → 編輯圖片 → 替代文字欄位填寫。

03 Functional Images 功能性圖片 🔴 跨專案經驗 🟡 常見高風險
▶

💡 白話說明

有些圖片不只是「圖」，而是「按鈕」或「連結」——比如點一個 Logo 會回到首頁，點一個放大鏡圖示會開搜尋。這類圖片的替代文字要說的不是「這張圖長什麼樣」，而是「按下去會發生什麼事」。

📋 技術規範

SC 1.1.1 Non-text Content (A) — 用作連結或按鈕的圖片，alt text 必須描述其功能（不是圖片內容）。

🔍 測試步驟

  • 找出所有可點擊的圖片和 Icon（Logo 連結、社群媒體圖示、Icon Button）
  • 在 Elementor 中確認這些圖片的 alt 描述的是動作/目的地（如「返回首頁」而非「箭頭圖示」）
  • Icon Widget / Icon Box Widget：檢查是否有設定 ARIA Label 或 Title
  • Social Icons Widget：Elementor 通常會自動處理，但需確認螢幕閱讀器朗讀內容
  • 圖片連結若旁邊已有文字連結，圖片的 alt 可留空（避免重複朗讀）
Icon Widget → Advanced → Attributes 可加入自訂屬性。但 Elementor 的 Icon Widget 本身對 ARIA 支援有限，需用螢幕閱讀器實際驗證。

04 Headings 標題結構 🔴 跨專案經驗 🟡 常見高風險
▶

💡 白話說明

標題就像書的目錄：大標題、中標題、小標題要有清楚的層級。視障者會用「跳到下一個標題」的方式快速瀏覽頁面。如果標題層級亂跳（像書的目錄從第一章直接跳到第三節），他們會迷路。

📋 技術規範

SC 1.3.1 Info and Relationships (A) + SC 2.4.6 Headings and Labels (AA) — 標題必須使用正確的標題層級（H1-H6），且層級不跳級。

🔴 跨專案經驗

在另一個評測專案中發現：標題層級跳級（H1 直接跳到 H3），或者只是為了讓文字看起來大/小而選錯標題層級。在 Elementor 中特別容易發生，因為 Heading Widget 的層級選擇和字體大小設定是分開的，容易混淆。

🔍 測試步驟

  • 在 Elementor 中每個 Heading Widget 都可選擇 H1-H6 層級
  • 關鍵：選擇層級要根據「結構邏輯」而非「視覺大小」
  • 每個頁面只有一個 H1
  • 層級不跳級：H1 → H2 → H3（不能 H1 → H3）
  • 如果想要小字的視覺效果，用 Elementor 的 Style 設定調字體大小，不要改標題層級
  • 使用 HeadingsMap 擴充功能檢查完整的標題結構樹
  • 螢幕閱讀器測試：用「H」鍵快速跳轉標題是否合理
Heading Widget → Content → Title → HTML Tag 下拉選單選擇 H1-H6。文字大小在 Style 分頁調整，與標題層級無關。

05 Page Structure & Landmarks 頁面結構與地標 🟡 常見高風險
▶

💡 白話說明

網頁的結構就像一棟房子：有門口（header）、走廊（導覽選單）、客廳（主內容區）、後院（footer）。如果程式有清楚標出這些區域，螢幕閱讀器就能讓使用者直接「跳到客廳」，不用從門口一步步走過去。

📋 技術規範

SC 1.3.1 Info and Relationships (A) + SC 2.4.1 Bypass Blocks (A) — 使用語意化地標區域（header, nav, main, footer）和 Skip Link。

  • 確認有 Skip to Main Content 連結（Ally 外掛可啟用此功能）
  • 用 Landmark 瀏覽器擴充檢查頁面地標是否完整
  • 螢幕閱讀器可用地標跳轉功能快速導覽
Ally 外掛設定中可啟用「Skip to Content」連結。在 Ally → Settings → Usability Features 中開啟。

06 Language 語言設定 🟡 常見高風險
▶

💡 白話說明

📋 技術規範

07 Text Contrast 文字對比度 🔴 跨專案經驗 🟡 常見高風險
▶

💡 白話說明

文字和背景的顏色差距要夠大，低視力的人才看得清楚。比如淺灰色背景配白色文字，一般人可能都覺得模糊——對視力不好的人更是完全看不到。這項用數字衡量：一般文字需要 4.5 倍以上的對比差距。

📋 技術規範

SC 1.4.3 Contrast (Minimum) (AA) — 一般文字對比度 ≥ 4.5:1，大文字（≥18pt 或 ≥14pt 粗體）≥ 3:1。

🔍 測試步驟

  • 使用 Colour Contrast Analyser (CCA) 滴管工具測試所有文字
  • 在 Elementor Site Settings → Global Colors 設定時就先驗證品牌色的對比度
  • 特別注意：圖片上的文字（Banner）需確保對比度或使用半透明背景遮罩
  • hover 和 focus 狀態的文字也需通過對比度
  • Elementor 的 Section/Container 背景色 + 文字色的組合都要檢查
在 Elementor Site Settings → Global Colors 設定品牌色系時，先用 CCA 確認每組前景/背景色組合都通過。在設計階段就把關，比事後修改容易得多。

08 Non-text Contrast 非文字對比度 🔴 跨專案經驗
▶

💡 白話說明

不只文字要夠清楚，按鈕、輸入框、圖示這些「非文字的東西」也要跟背景有足夠的顏色差距。比如一個白色背景上放一個淺灰色邊框的輸入框，低視力的人可能根本不知道那裡有個輸入框。

📋 技術規範

SC 1.4.11 Non-text Contrast (AA) — UI 元件（按鈕邊框、表單欄位邊框、icon）和圖形物件的對比度 ≥ 3:1。

🔍 測試步驟

  • 用 CCA 測試 Elementor Button Widget 邊框/背景與周圍背景的對比度
  • Elementor Form Widget 的 input 邊框與背景的對比度
  • Icon Widget 與背景的對比度（各狀態：normal, hover）
  • Focus indicator 的對比度
  • Ally Widget 本身的對比度也需確認
在 Elementor Button Widget → Style 中，確保按鈕的 Normal 和 Hover 狀態都通過對比度。Form Widget → Style → Field 中確認 input 邊框色。

09 Links 連結 🔴 跨專案經驗 🟡 常見高風險
▶

💡 白話說明

連結的文字要讓人一看就知道「點了會去哪裡」或「點了會發生什麼」。如果頁面上有五個「了解更多」，螢幕閱讀器使用者完全不知道哪個是哪個。另外，連結不能只靠「藍色」來表示它是連結——色盲的人看不出來。

📋 技術規範

SC 2.4.4 Link Purpose (In Context) (A) + SC 1.4.1 Use of Color (A) — 連結文字必須有意義，且不能僅靠顏色區分連結。

🔍 測試步驟

  • 所有 Button Widget / 文字連結的文字在上下文中可理解其目的
  • 避免「按此」「更多」「了解更多」— 改為「查看 2026 高峰會議程」「閱讀無障礙政策」
  • 連結不能僅靠顏色區分：需有底線或其他視覺提示（在 Elementor Typography 設定中加底線）
  • 新窗口開啟的連結需提示使用者（Elementor Link 設定中可選「Open in new window」，但需在連結文字中註明）
  • 螢幕閱讀器「Links」清單中，連結文字是否可單獨理解
Button Widget → Content → Link 設定連結目標。Text Editor Widget 中的連結在視覺模式下直接插入。如需加入 ARIA label，可在 Advanced → Attributes 中設定。

10 Forms 表單 🟡 常見高風險
▶

💡 白話說明

每個輸入框旁邊都要清楚標示「這裡要填什麼」，而且螢幕閱讀器也要能唸出來。想像你閉著眼睛填表格——如果沒有人告訴你「這格是姓名」「那格是 email」，你完全無從填起。另外，同一個表單流程不要重複問已經填過的東西。

📋 技術規範

SC 1.3.1 (A) + SC 3.3.2 Labels or Instructions (A) + SC 4.1.2 Name, Role, Value (A) — 表單欄位必須有 label，且 label 與欄位關聯。

🆕 WCAG 2.2：SC 3.3.7 Redundant Entry (A) — 同一流程中不重複要求輸入相同資訊。

🔍 測試步驟

  • Elementor Form Widget 的每個欄位都要填寫 Label
  • 必填欄位在 Form Widget 設定中勾選 Required
  • Placeholder 不能替代 Label（Elementor 允許隱藏 Label 只顯示 Placeholder，這對無障礙不利）
  • 錯誤訊息是否清楚指出哪個欄位有問題
  • 🆕 多步驟表單不重複要求相同資訊
  • 用鍵盤 Tab 能完整操作表單（填寫、選擇、送出）
Form Widget → Form Fields → 每個欄位設定 Label、Placeholder、Required。注意不要勾選「Hide Label」除非有其他方式標示欄位用途。
Elementor Form Widget 的無障礙支援並非完美。特別注意：錯誤訊息的關聯、群組欄位的結構，可能需要額外用 Ally 掃描來確認。

12 Keyboard Navigation 鍵盤導覽 🟡 常見高風險
▶

💡 白話說明

有些人無法使用滑鼠（例如手部障礙），只能用鍵盤的 Tab 鍵在網頁上移動。所有按鈕、連結、選單都必須能用鍵盤操作到，而且不能讓鍵盤「卡住」出不來（例如打開一個彈出視窗後按什麼鍵都關不掉）。

📋 技術規範

SC 2.1.1 Keyboard (A) + SC 2.1.2 No Keyboard Trap (A) + SC 2.4.3 Focus Order (A) — 所有功能可用鍵盤操作，且不會困住焦點。

🆕 WCAG 2.2：SC 2.4.11 Focus Not Obscured (AA) — 焦點不被 sticky 元素完全遮擋。

🔍 測試步驟

  • 從頁面頂部開始按 Tab，驗證所有互動元素都可到達
  • Tab 順序符合視覺順序（左到右、上到下）
  • Elementor Nav Menu Widget 的下拉選單能否用鍵盤展開和關閉
  • 如有 Popup / Modal，焦點是否正確移入和移出
  • Accordion / Toggle Widget 能否用 Enter 鍵展開/收合
  • 🆕 Tab 時焦點不會被 Elementor Sticky Header 遮擋
  • Carousel / Slider Widget 能否用鍵盤操作
Elementor 某些 Widget（如 Accordion）已知有鍵盤操作問題。Tab 測試時要特別注意這些 Widget，如發現無法用鍵盤操作，記錄為平台限制並尋找替代方案。

13 Focus Visible 焦點可見性
▶

💡 白話說明

用鍵盤在網頁上移動時，你需要知道「現在游標在哪裡」。通常是一個方框或底線標示出目前選到的東西。如果網頁把這個標示拿掉了（很多人覺得不好看），鍵盤使用者就像在黑暗中摸索。

📋 技術規範

SC 2.4.7 Focus Visible (AA) — 鍵盤焦點必須有清晰的視覺指示器。

🆕 WCAG 2.2 (AAA)：SC 2.4.13 Focus Appearance — focus indicator 至少 2px 外框、對比度 ≥ 3:1（雖為 AAA，建議達成）。

🔍 測試步驟

  • Tab 遍歷所有元素，確認每個都有可見的 focus 指示器（方框、底線、色彩變化）
  • 確認主題或 Elementor 設定沒有移除預設焦點樣式
  • Focus indicator 不能被其他元素遮擋（WCAG 2.2 新要求）
  • 在不同背景色的 Section 上測試 focus 可見性
Elementor 對 focus 樣式的控制有限。Focus 樣式主要由 WordPress 主題的 CSS 決定。如果預設 focus 不夠明顯，可能需要在主題的 Custom CSS 中加入 focus 樣式（這是 no-code 的邊界情況，與 Christina 討論處理方式）。

15 Resize & Reflow 文字放大與重排
▶

💡 白話說明

低視力的人需要把文字放很大才看得到。放大之後，文字不能跑出畫面、不能互相疊在一起、不能需要左右滑動才看得完。簡單說：用 Ctrl + 放大到 200%，整個網頁看起來還是正常的就對了。

📋 技術規範

SC 1.4.4 Resize Text (AA) + SC 1.4.10 Reflow (AA) + SC 1.4.12 Text Spacing (AA) — 文字可放大至 200% 不失去內容；320px 寬度不出現水平捲軸。

  • Elementor 的 Desktop / Tablet / Mobile 響應式設定是否正確
  • 確認 Elementor 字型設定使用相對單位而非固定 px
Elementor 的 Responsive Mode（底部切換 Desktop/Tablet/Mobile）可針對不同斷點調整字體大小和版面。確保在 Mobile 模式下所有內容都正常顯示。

▶

18 Navigation Consistency 導覽一致性
▶

💡 白話說明

選單在每一頁都要長一樣、放在一樣的位置。如果首頁的選單是「關於 | 活動 | 聯絡」，其他頁面也要是這個順序。使用者（尤其是認知障礙者）一旦學會了選單位置，換一頁就不應該要重新找。

🆕 WCAG 2.2：SC 3.2.6 Consistent Help (A) — 幫助機制在所有頁面位置一致。

  • 相同功能在不同頁面使用相同名稱和 icon

  • Footer 連結在所有頁面一致

▶

💡 白話說明

每個頁面最上面都有選單，如果你用鍵盤操作，每次都要按好幾十下 Tab 才能跳過選單到達主要內容，非常痛苦。「跳過導覽」連結讓鍵盤使用者按一下就能直接跳到主要內容。平常看不到它，但用 Tab 按到時它會出現。

💡 白話說明

有些圖片純粹是裝飾用的（比如底圖花紋、分隔線裝飾），不傳達任何資訊。這類圖片要讓螢幕閱讀器跳過，不然使用者會聽到一堆「圖片、圖片、圖片」卻不知道是什麼。判斷方法：把圖片刪掉，如果不會少了什麼資訊，就是裝飾性圖片。

🔍 測試步驟

  • 裝飾性圖片：在 Elementor Image Widget 的 Alt Text 欄位留空
  • 使用 Elementor Section/Container 的背景圖片功能放裝飾圖（背景圖預設不會被螢幕閱讀器讀取）
  • 螢幕閱讀器不應朗讀裝飾性圖片

23 🆕 Target Size 目標大小 WCAG 2.2 新增
▶

💡 白話說明

按鈕和可點擊的東西不能太小。想像在手機上用大拇指去按一個超級小的 X 按鈕——很容易按錯。WCAG 2.2 新規定：所有可點擊的東西至少要 24×24 像素大（大約一顆綠豆的大小）。更好的做法是做到 44×44 像素。

📋 技術規範

SC 2.5.8 Target Size (Minimum) (AA) — 指標輸入目標至少 24×24 CSS 像素。

🔍 測試步驟

  • 用瀏覽器 DevTools 量測 Elementor Button Widget 的實際點擊區域
  • Elementor Button Widget → Style → Padding 可控制按鈕大小，確保至少 24×24px
  • Nav Menu Widget 的選單項目間距需足夠
  • Icon Widget / Social Icons Widget 的大小需足夠
  • 特別注意 Mobile 模式下的觸控目標（在 Elementor Responsive Mode 中檢查）
  • 建議：做到 44×44px 更好
Button Widget → Style → Padding 增加內距可有效增大點擊區域。Icon Widget → Style → Size 可調整 Icon 大小。在 Mobile 響應式設定中可另外調整。

24 🆕 Dragging Movements 拖曳替代 WCAG 2.2 新增
▶

💡 白話說明

有些功能需要「拖曳」才能操作（比如滑動輪播圖片）。但有些人無法做出拖曳動作。所以每個拖曳功能都要有替代方式——比如輪播圖片旁邊加上「上一張」「下一張」的按鈕。如果網站沒有任何拖曳功能，這項就自動通過。

📋 技術規範

SC 2.5.7 Dragging Movements (AA) — 所有拖曳功能必須有替代的單點擊操作。

🔍 測試步驟

  • 檢查是否使用 Carousel / Image Carousel / Slides Widget
  • 這些 Widget 是否有顯示箭頭按鈕（上一個/下一個）和分頁點
  • 在 Elementor Carousel Widget → Content → Navigation 確認啟用 Arrows 和 Dots
  • 嘗試只用鍵盤完成輪播操作
  • 如果沒有拖曳功能，此項自動通過
Carousel Widget → Content → Additional Options → Navigation 設定中，確保 Arrows 和 Pagination 都設為「Show」。

25 🆕 Accessible Authentication 無障礙認證 WCAG 2.2 新增
▶

💡 白話說明

登入的時候不能強迫使用者做「動腦筋的測試」（比如記住密碼、解數學題、從九宮格裡選出紅綠燈的圖片）。認知障礙的人做不了這些事。要讓人可以用密碼管理器自動填入，或提供 Google 登入之類的替代方式。如果網站根本沒有登入功能，這項不用管。

📋 技術規範

SC 3.3.8 Accessible Authentication (Minimum) (AA) — 認證流程不能依賴認知功能測試，除非有替代方式。

🔍 測試步驟

  • Phase 1（Home + Events）如無會員登入功能，此項不適用
  • 如有表單含 CAPTCHA：確認使用無障礙友善的方案（如 hCaptcha、reCAPTCHA v3 隱形版）
  • Elementor Form Widget 若整合 reCAPTCHA：在設定中選擇 v3（不需使用者操作）
  • 確認密碼欄位支援 paste（不阻擋密碼管理器）
Elementor Form Widget → Content → 可新增 reCAPTCHA 欄位。在 Elementor → Settings → Integrations 中設定 reCAPTCHA 金鑰。建議使用 v3（隱形驗證），避免使用需要使用者手動操作的 v2 勾選框。
