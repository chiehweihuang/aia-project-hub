3 分鐘讀完:AIA 收款方案結論

一句話結論:現在就採 netiCRM 微型「最小配置」(NT$12,960/年)。這一筆預算同時解決兩件事 — 現在:讓 ADALS 2026 能線上刷卡售票(海外與會者尤其需要);未來:AIA 變成社團法人那天,抵稅收據等合規功能在同一個工具裡直接開通,不換系統、不搬資料。

Recur.tw 本身沒有活動報名功能,單獨不能當 ADALS 的售票工具。但有一個組合值得評估:WordPress 報名 plugin + Recur 當金流(報名交給 plugin、收款交給 Recur)。這條路可不可行,取決於整合乾不乾淨、會不會變成跨系統對帳 — 是諮詢 Recur 的重點,問題清單見 諮詢準備頁 的 R-C 區。

先看清楚:AIA 現況怎麼影響這個決定

AIA 目前是人民團體,還不是社團法人,也還沒拿到所得稅法 §11(4) 認定。意思是:AIA 現在開不了抵稅收據 — 這跟用哪個軟體無關,是法律身分的問題。

所以一個常見的誤解要先澄清:「選 netiCRM 是因為它有抵稅收據功能」這個理由,現階段其實還不成立 — 那個功能現在是 dormant(在,但用不到)。同理,公益勸募合規、企業捐款統一編號,現在也都用不到(企業捐款本來就走手動 invoice + 銀行匯款,不需要系統,見下方「對 AIA 的影響」區)。

現階段真正需要 netiCRM 的理由只有一個:ADALS 2026(150-200 人、多票種、有海外與會者)需要線上刷卡售票。而能做「線上售票 + 單一可維護系統」的,現實上的底線就是 netiCRM 這一類工具(KKTix / Accupass 可及性已排除,Eventbrite 費率與台灣 payout 已排除;表單工具 + 金流連結的拼裝法違反「不跨系統對帳」原則)。

為什麼「最小配置 netiCRM」就是兩全其美

關鍵洞察:netiCRM 微型方案 NT$12,960/年 這個價格,不會因為合規功能現在用不到就比較貴。你付一樣的錢,合規功能等 AIA 變社團法人那天自動開通。所以「現在便宜」和「未來不用搬家」不是二選一:

「先省錢用手動、以後再換」為什麼不划算:Google Form + ATM 轉帳不是線上刷卡 — 與會者要自己去轉帳、AIA 要人工核帳、海外與會者根本沒辦法做台灣轉帳。ADALS 200 人用手動估 50-70 小時人工核帳工時,海外還得另接 PayPal(4.4%),加上報名 drop-off。而且 AIA 未來變社團法人時,還是得搬到 netiCRM,付一次搬家成本。結論:手動沒有比較省,只是把成本從「現金」換成「人力 + 未來搬家」。(手動仍適合小型免費 / 低價活動,見下方分階段路徑)

情境 A:純小額月捐 program — 月營收必須 ≤ NT$160K(不需抵稅收據)

SCENARIO A · 月捐

前提條件(全部成立才省錢):

  • AIA 推出小額月捐 program(NT$100-500/月)
  • 月營收 ≤ NT$160K/月(超過此線 Recur 邊際 4.9% > 藍新 2.5%,反而比 netiCRM 貴)
  • 目標捐款者不在意抵稅收據(例如:小額情感性支持、信仰群體、不報稅者)
  • 單一付款方式即可(信用卡 recurring,不需 ATM / 超商代碼)
  • 不需要 donor CRM 後續分眾 / 寄信 / campaign tracking

實際省多少:NT$50K/月 Recur 省 NT$1,130/月;NT$100K/月 省 NT$1,180/月;NT$200K/月 反轉,netiCRM 省 NT$1,220/月。Recur 的省錢窗口是月捐規模 NT$160K 以下,且 donor 不在意抵稅。

情境 B:Creator economy 副業產品線

SCENARIO B · 課程 / 訂閱

前提條件:

  • AIA 推付費課程 / 付費白皮書 / 付費社群(Discord / Circle / Slack)
  • 這些是商業性收入(不算捐款) → 不需 NPO 抵稅收據,只需開統一發票或一般收據
  • 跟 NPO 主業務(捐款 / 會員 / event)會計獨立切分(避免混雜課稅)

實際省多少:這條收入線本來就不適合放 netiCRM。Recur 接這塊 vs 自建 Stripe + WordPress + Memberships plugin 第一年 NT$30-50K + 年費 NT$10-20K。Recur 在此情境結構性勝出,且跟 netiCRM 並用不互斥。

情境 C:Pre-11 條認定階段過渡期(2026 全年可能適用)

SCENARIO C · 過渡期

前提條件:

  • AIA 目前是人民團體 status,還沒拿 11 條認定 + 公益勸募字號
  • 意思:即使用 netiCRM,當下也還不能開抵稅收據(等認定批准才 unlock)
  • 因此「Recur 沒抵稅收據」這個 deal-breaker 在這段過渡期暫時是 moot
  • 過渡期預估 6-12 個月(申請 11 條認定 + 內政部審查時程)

實際省多少:過渡期每月 NT$100K 營收下,Recur 比 netiCRM 微型 + 藍新省約 NT$1,180/月。但過渡期一過,netiCRM unlock 抵稅 → advantage 反轉,且當時切回 netiCRM 也要重建 stack。除非有信心 11 條認定永遠不會通過,否則這個 saving 是「短多長空」。

情境 D:0 預算 bootstrapping 階段(極小規模試水溫)

SCENARIO D · 0 預算

前提條件:

  • AIA 第一年完全沒預算編 SaaS 年費
  • 月營收 < NT$50,000(小型試辦活動 / 少量月捐)
  • 能接受手工開收據(不用系統化)
  • 未來 1-2 年內不打算規模化

實際省多少:0 固定成本是 Recur 強項。但 0 預算階段更該考慮純免費 stack(Google Form + 銀行匯款 + 手工收據),連 Recur 2.4% 都省掉。Recur 在這情境只贏免費 stack 的「自動化便利度」,不贏「成本」。

情境 E:大規模國際付款(高頻、無 NPO 法規)

SCENARIO E · 國際付款

前提條件:

  • AIA 接大量海外個人付費(訂閱、課程、會議票)
  • 海外付款者不在意抵稅(本來也抵不了台灣稅)
  • 需要訂閱式自動續費(non-event 收入)

實際省多少:對海外付款者,Recur 的訂閱 dunning UX 比 netiCRM 強。但 Recur 是否真接得到海外卡 + 跨幣別,本頁未驗證,要問 Recur 客服。如果 Recur 海外收款不順,這情境直接 fallback 到 PayPal Business + Stripe 路徑。

整合判斷:
  • 情境 B(creator economy 副業)是 Recur 唯一「結構性勝出」的情境。可考慮 netiCRM(主)+ Recur(副)雙軌。
  • 情境 A / C / E是「條件性勝出」 — 前提條件全部成立才算省錢,任一條不成立就反轉。
  • 情境 D不是 Recur 強項 — 0 預算純手工在這情境更省。
  • 所有情境共通的未驗證假設:Recur 是否真不能開 NPO 抵稅收據(可能本頁論述過於斷定,建議另外諮詢確認)。

理由

  • ADALS 2026 event 規劃以 netiCRM iframe 為入口,Recur 不支援 event registration = 改採 Recur 等於從零重建
  • AIA 取得所得稅法 §11(4) 認定後,netiCRM 解鎖完整 NPO compliance — Recur 依公開定位看似不在此 product 範圍
  • Recur 並非「全包 2.4%」— 月營收 > NT$100K 後邊際 4.9%,中大型 NPO 反而貴
  • netiCRM 微型方案 NT$12,960/年 對小 NPO 是 trivial cost;最小配置可進一步壓低

只有兩個情境會讓 Recur 成有 ROI 的 add-on

  1. AIA 推付費課程 / 白皮書訂閱 / 付費社群 → Recur 可能 supplement netiCRM(creator 那一塊用 Recur,NPO compliance 仍 netiCRM)
  2. AIA 月捐 program churn 嚴重 + Recur 確認支援 NPO 收據開立 → Recur 補 dunning UX

兩個情境前提都成立才考慮 Recur,目前 AIA 都不在這 trajectory。

一個更深的觀察 — 決策框架差異

具科技 / SaaS 產品背景的決策者,容易高估「UI 現代化」的權重、低估「法規合規」的權重。這 instinct 在 commercial product 是對的,但在 Taiwan NPO compliance 場域需要切換 frame:

  • Commercial product:UX / API / 現代化 > 法規合規(可以後補)
  • NPO operations:法規合規 = donor trust = revenue。UX 老派可接受,合規不能省

netiCRM UI 老派但合規完整;Recur UI 現代但 NPO 法規層待驗證。對 NPO operator,本頁建議優先合規而非美觀 — 但這個 trade-off 是 explicit 的,留給理事會權衡。

  • 沒 vendor escalation channel

  • 工程師離職的交接費:程式沒有文件,新工程師要從頭看懂
  • 每加新功能都要請工程師:第二屆 ADALS 加早鳥規則、退款等都要再付開發費
  • 緊急修理費:出事時緊急叫人比平時貴 2-3 倍
共同的 per-transaction 費用(三條一樣):
  • 國內刷卡:藍新 2.4%(可申請)/ 2.75%(標準)
  • 國際刷卡:PayPal 4.4% + USD$0.30/筆
  • 跨境贊助:SWIFT 手續費 USD 40-80/筆

  • Price(預付 + 年費):D(0)< A < B << C(高一個量級)
  • Effort + Risk(對 ADALS 規模 200 人):A < B << D < C

  • Effort + Risk(對小型活動 < 30 人):D < A < B << C

→ Path D 是條件性選項:不適用 ADALS 規模,但適合 APAC AIA 未來辦小型 workshop / community 聚會時的 0 成本 baseline。

§5 WHY NETICRM

為什麼 netiCRM 是主推

三層結構性整合(Eventbrite 處理不了的)

  1. 台灣稅務無縫對接 — netiCRM 內建符合台灣稅法的收據格式(統編、捐款扣抵編號、社團法人格式),NPO 法人會計直接可用。Eventbrite 給的是美式收據,要另外手動補開台灣格式收據 — 雙重工時 + 合規風險。
  2. 海外付款者整合 — netiCRM 內建國際卡(底層藍新)讓海外與會者用本地信用卡付款;若需要 PayPal UX,可另加 PayPal Business 作為 alternative checkout。後台統一進 NPO 帳戶並產生對應收據。Eventbrite 的 Taiwan organizer payout 不在 Pro 支援名單。
  3. 會員資料跨年度持續使用 — 會員年費、贊助履約追蹤、捐款歷程、自動寄信、會員資格自動審查 — 這些 NPO 標準工作流程 netiCRM 內建,Eventbrite 沒有(它是售票平台,活動結束資料就斷了)。

§7 PAYMENT DIFF

跨國 / 線下 / 線上 / Hybrid 付費差異

同樣的活動規模,不同的舉辦形式 + 觀眾來源,會直接影響付款基礎建設需求:

維度 線下 (Offline) 線上 (Online) Hybrid 跨國 (International)
主要金流 線上預付 + 現場補單(現金/轉帳) 100% 線上(無現場退路) 兩條並行,需各自 SKU 必須 PayPal / 國際卡(NewebPay 不支援國際付款者)
票種結構 1–2 種(早鳥 / 一般) 通常 1 種 至少 2 SKU(現場 / 線上) 多幣別 SKU(USD / TWD)
收據格式 可選擇實體 / 數位 數位 only 兩種選項並存 必須英文(統編改 Tax ID)
容量管理 場地硬上限 幾乎無限(看頻寬) 兩條容量分管 取決於線上 / 線下混合比例
退費處理 涉及實體成本(已定餐、已印名牌) 簡單(數位資產可回收) 兩條政策可能不同 跨境匯費吃成本(USD 50+ 不退費)
不需要 國際卡(若只給台灣) 場地金流硬體 純現場票流程 純台灣 NewebPay(對國際付款者無感)
特別風險 現場報到混亂、現金管理 連結漏寄、Zoom session limit 兩邊體驗不一致(線上者覺得二等) 時區、語言、信用卡跨境反詐騙拒絕
關鍵 takeaway:付費複雜度真正的驅動因子是「觀眾的地理範圍」+「活動形式」這兩個正交軸。線下台灣活動最簡單(只需藍新);跨國 Hybrid 活動最複雜(藍新 + PayPal + SWIFT + 雙語收據都要)。設計平台時要先確認這兩軸,再決定金流 stack。
