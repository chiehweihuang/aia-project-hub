§1 OVERVIEW

範本目的

ADALS 2026 不是一般的會議網站。這場會議要面對 CTO 級觀眾，網站本身就是 WCAG 2.2 AA 的範例。這份範本聚焦在最容易出問題的兩個地方：報名表單與付款流程。

四種收款情境（出自 §6.1）

不能用同一個工具解決所有情境。不同情境走不同路徑：

台灣付費門票
NT$1,500–3,000 · 個人 · 集中
藍新金流（信用卡 + ATM + 超商）
國外付費門票
USD 60–120 · 個人 · 少量
PayPal Business（信用卡）
台灣企業贊助
NT$50K–300K · 公司 · 一次性
玉山銀行 + Invoice（不上線收）
國外企業贊助
USD 1,500–10,000 · 公司 · 一次性
SWIFT 電匯 + 英文 Invoice
關鍵設計原則：贊助走 invoice + 匯款，門票走線上刷卡。網站只負責「門票」流程，贊助是「洽詢表單 → 人工開立 invoice → 線下匯款」三步走，不上網路即時收款。

為什麼五步驟而不是一頁式？Stripe Atlas、Google Forms 等研究都指出：付費表單分步可以降低中途放棄率（特別是付款卡關時，使用者不會丟掉前面填好的資料）。但分步必須有清楚的進度指示和可回到上一步的能力，否則會造成挫折。
§3 STEP 1 · TICKET SELECTION

票種選擇示範

DEMO · STEP 1 / 5
選擇票種
設計重點
  • <fieldset> + <legend> 包整組 radio，screen reader 會把整組讀為一個問題
  • 整張卡片可點，但真實 input 是 radio，視覺只是 wrapper
  • 選中狀態用 border + background 雙重提示，不只靠顏色（色盲使用者）
  • 價格用 IBM Plex Sans 等寬字體，數字易辨識
  • 票種代碼（STANDARD/EARLY BIRD）用全英文大寫 + tracking，是視覺錨點，不是給螢幕閱讀器念的
  • 學生票要明確寫「須上傳學生證」，避免事後爭議
  • 國際票特別標明「海外付款 / PayPal」，告知這條會走不同付款流程
§4 STEP 2 · REGISTRATION FORM

報名表單示範

ADALS 報名表單應該盡量短，只問付款與通知絕對需要的資訊。其他偏好（飲食、住宿、議程關注）可以在報名成功後用後續 email 收集，不要塞進報名流程。

DEMO · STEP 2 / 5
將出現在報到名牌上。
電子郵件格式不正確，請輸入完整地址（例如 name@domain.com）。
用於議程後分組討論時的名牌資訊。
幫助主辦方安排議程互動分組。
無障礙需求（選填）
每個欄位都有理由
  • 姓名：報到名牌
  • Email：付款收據、議程提醒、無障礙需求確認
  • 服務單位：分組討論名牌（選填，避免無組織者卡關）
  • 角色身分：議程互動安排（5 選項即可，不要過細）
  • 無障礙需求：用 fieldset 包，避免被誤認為單選
a11y 重點
  • 每個 input 必須有 <label for>
  • 必填欄位用 aria-required="true"（不要只靠紅色 *）
  • 錯誤訊息用 aria-invalid + aria-describedby 連結到 role="alert"
  • autocomplete 屬性讓密碼管理工具與輔助技術可以填入正確內容（WCAG 1.3.5）
  • iOS Safari：input 必須 font-size: 16px 或以上，避免 focus 時自動放大頁面
不要犯的錯：用佔位符（placeholder）取代 label。Placeholder 在輸入後消失，老人和認知障礙使用者會忘記這個欄位是什麼。Label 必須一直在。
§5 STEP 4 · PAYMENT BRANCHING

付款方式分流

付款不是一個畫面，是一個分流路由器。根據 Step 1 的票種選擇，導向不同金流：

票種 → 付款方式
一般票 / 早鳥 / 學生票 → 藍新金流 NewebPay（信用卡 / ATM / 超商代碼）
國際票（USD） → PayPal Business（信用卡，國際付款）
企業贊助洽詢 → 不上線收，跳轉到贊助洽詢表單，由 Claire 人工開立 invoice + 等銀行匯款

分流 A：藍新金流（台灣門票）

MOCK · NEWEBPAY EMBED
藍新金流付款頁
實際是跳轉到藍新代收頁面，由藍新處理刷卡 / ATM / 超商代碼三種付款。
付款完成後 callback 回 ADALS 網站確認頁。
POST → core.newebpay.com/MPG/mpg_gateway
為什麼是藍新
  • 信用卡分期一般會員就能用（綠界要付 NT$10,000 年費）
  • 費率可申請 2.4%（綠界 2.75%）
  • 子公司 ezPay 是唯一電子支付專營執照，合規位階高
  • API 結構與綠界類似，技術整合難度相當
a11y 注意
  • 跳轉前要明確告知「您將被導向藍新付款頁」
  • 付款失敗回來時要保留前面填的資料，不要清空
  • callback 確認頁要有清楚的成功 / 失敗訊息，不要只寫 200/400

分流 B：PayPal Business（國際門票）

MOCK · PAYPAL CHECKOUT
PayPal 結帳
使用 PayPal Smart Buttons SDK 嵌入。使用者可以用 PayPal 帳戶或信用卡付款（不需註冊 PayPal 帳號）。
領款匯入 APAC AIA 玉山銀行帳戶。
<script src="paypal.com/sdk/js?client-id=...">
限制
  • 費率 4.4% + USD 0.30（高於藍新，但這是台灣協會收國外刷卡唯一現實選項）
  • 必須先開玉山銀行帳戶，才能把 PayPal 餘額領出來
  • 已關閉台幣交易（剛好符合本場景：只用於外幣）
不要選的東西
  • Stripe：2026/04 仍不支援台灣本地實體
  • Wise Business：台灣協會明確不在支援名單
  • Payoneer：NPO 法人支援不明朗

分流 C：贊助洽詢（不走線上付款）

為什麼不上線收？
  • 金額大，企業財務需要 invoice + 匯款 audit trail
  • 線上刷卡費率（2.4–4.4%）對 NT$300K 是 NT$7,200–13,200 額外成本
  • 企業贊助節奏慢，不需要即時收款
§6 SPONSOR INQUIRY

贊助洽詢流程（與門票分開）

贊助流程的目的是收集足夠的企業資訊，讓 Claire 能直接開立 invoice 並在後續對話中討論細節。線上不收錢、不簽合約，只收意向。

DEMO · SPONSOR INQUIRY FORM
用於開立 invoice。國外公司請填 Company Registration Number。
USD 將提供 SWIFT 資訊，匯款手續費由付款方負擔（國際慣例）。
表單欄位邏輯
  • 贊助級別放第一個，讓不確定的人能選「想了解更多」進入對話
  • 統一編號 / Tax ID 一個欄位涵蓋台灣與國外，hint 說明差異
  • 付款幣別偏好讓贊助方一開始就知道有 SWIFT 選項，不會以為只能 TWD
  • 不問「您希望獲得什麼回饋」（這在贊助方案頁已寫清楚）
  • 不問「您想了解什麼」（送出後 Claire 會直接打電話 / 寫信）
送出之後
  • 顯示「已收到，Claire 會在 2 個工作日內回覆」
  • 立刻寄一封確認 email，附上贊助方案 PDF
  • 不要說「請等通知」這種被動語
§7 STATES

狀態示範

Elementor 預設只處理「正常」狀態。但實際使用時，使用者會遇到很多非正常狀態。每個狀態都需要明確的視覺與文字提示。

LOADING

處理中

  正在送出您的報名⋯

spinner 用 aria-hidden="true"（純裝飾），實際狀態用 role="status" 或 aria-live="polite" 文字告知 SR。
尊重 prefers-reduced-motion，停止旋轉動畫。
ERROR

表單驗證失敗

⚠ 請檢查以下欄位：電子郵件格式不正確。

錯誤摘要放在表單頂端並 focus 過去；個別欄位錯誤用 role="alert" + aria-describedby。
不要只用紅色，必須有圖示或文字。
PAYMENT FAILED

付款失敗

付款未成功，您的資料已保留。可以：

  • 重新嘗試付款
  • 更換付款方式
  • 聯繫主辦方協助
關鍵：絕對不要清空表單資料。失敗訊息要說清楚「為什麼失敗」（卡片被拒、餘額不足、超時等）。
SUCCESS

報名成功

✓ 報名完成！報名編號：ADALS-2026-00142

  • 確認信已寄至 your@email.com
  • 議程 PDF 連結 5 分鐘內寄達
  • 無障礙需求由專人回覆
成功頁應該立刻可以做下一件事（加入行事曆、分享、加入 LinkedIn 群組），而不是 dead end。
容易被忽略的狀態：
  • 網路斷線：表單送出失敗時要有重試機制，不能直接拋錯
  • 學生票售完：在票種選擇頁就要把該選項標記為「售完」，不要送出後才報錯
  • session 過期：填一半離開回來，要能還原（用 localStorage 暫存）
  • 付款處理中重複按鈕：按鈕送出後要立刻 disable，避免重複扣款
§8 TECH NOTES

技術註記

表單服務選擇

建議

  • Fillout（fillout.com）— 有 WCAG 公開承諾，可嵌入 iframe，logic branching 支援良好
  • Cognito Forms（cognitoforms.com）— 有 a11y 聲明，付費版可串金流
  • 若 Christina 想用 Elementor 原生表單：用 Elementor Pro Forms + Ally 外掛 補強，不要用 Contact Form 7（a11y 不及格）

不要用

  • Typeform — 動畫過度、鍵盤導航有缺陷、screen reader 體驗差
  • KKTix / Accupass — 內部評估可及性不可接受
  • Eventbrite — 跨境費率累計 6–8%、版型限制論述呈現
  • Google Forms — 無法嵌入金流，且樣式無法符合 ADALS 設計風格

WCAG 2.2 AA 表單檢查清單

  • 所有 input 有 <label for>，不用 placeholder 取代 label（1.3.1, 3.3.2）
  • 必填欄位用 aria-required="true"（不只靠紅色 *）
  • 錯誤訊息用 role="alert" 或 aria-live="assertive"（3.3.1）
  • 錯誤訊息明確指出是什麼錯 + 怎麼修正（3.3.3）
  • autocomplete 屬性正確使用（1.3.5）
  • Radio / Checkbox 群組用 <fieldset> + <legend>
  • 所有互動元素 focus 可見（2.4.7），不要 outline: none
  • 觸控目標 ≥ 44×44 CSS pixels（2.5.8）
  • iOS 防自動縮放：input font-size: 16px 以上
  • 表單可用鍵盤完整操作（2.1.1）
  • 送出按鈕避免雙擊重複送出（disable on submit）
  • 顏色對比 ≥ 4.5:1（文字）、3:1（UI 元件邊框）

雙語切換

使用 data-zh="..." / data-en="..." 屬性，JS 切換時必須同步更新 <html lang> 屬性（讓 screen reader 用對的語言發音）。Form label、error message、button text 都要支援雙語切換。

整合工具評估：netiCRM

強烈建議 Claire 先諮詢 netiCRM（neticrm.tw），它是專為台灣 NPO 設計的 CRM，內建報名、收款、會員管理、收據開立、金流串接（綠界、藍新都已整合，且新客戶送一家金流免費導入）。如果 netiCRM 適合，後續金流串接、會員系統、收據都一站解決，Christina 就只需要把 netiCRM 的報名頁嵌入 ADALS 站。

不要做的事

  • 不要先做網站再想金流（金流申請時間會反向影響網站設計）
  • 不要假設 Stripe 可開（2026/04 仍不行）
  • 不要假設 Wise Business 可開（台灣協會明確不在支援名單）
  • 不要用 Tailwind 預設樣式
  • 不要用 Inter / Roboto / 系統 sans-serif 作為主要字體
  • 不要把內部金流附錄放在 public 網站（前一個 session 的錯誤）
  • 不要用 stock photo（破壞論述質感）
