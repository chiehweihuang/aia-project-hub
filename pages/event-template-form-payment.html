文件定位

PART 1 · §B · 可及性最低門檻

可及性最低門檻(WCAG 2.2 AA)

主要清單:本專案 WCAG 2.2 AA 測試清單以 testing-flow.html 為 canonical 來源(34 個 step,涵蓋全頁測試)。實作前請以該頁為準。 本節只列出對「報名 / 付款表單」特別重要的子集 + 範本級補充。

從 testing-flow 取出的 form 相關子集

這些是 testing-flow 中對表單 / 付款流程影響最大的條款,實作時優先檢查:

  • 1.3.1 Info and Relationships(A)— 表單欄位有 <label>,錯誤訊息與欄位用 aria-describedby 關聯
  • 1.4.1 Use of Color(A)— 錯誤訊息不能只靠紅色 / 必填不能只靠紅星號
  • 1.4.11 Non-text Contrast(AA)— input 邊框、focus ring 對背景 ≥ 3:1
  • 2.1.1 Keyboard(A)— 報名流程可純鍵盤完成
  • 2.4.3 Focus Order(A)— 欄位 Tab 順序合理
  • 2.4.7 Focus Visible(AA)— focus ring 明顯可見
  • 2.4.11 Focus Not Obscured(AA, WCAG 2.2 新)— sticky header / modal 不遮焦點
  • 2.5.8 Target Size (Minimum)(AA, WCAG 2.2 新)— 互動元件 ≥ 24×24 px,建議 44×44
  • 3.3.1 Error Identification(A)— 指出哪個欄位錯
  • 3.3.2 Labels or Instructions(A)— 可見 label,不只 placeholder
  • 3.3.3 Error Suggestion(AA)— 提供修正建議
  • 3.3.7 Redundant Entry(A, WCAG 2.2 新)— 多步驟不重問同樣資訊
  • 3.3.8 Accessible Authentication(AA, WCAG 2.2 新)— 若有登入,支援密碼管理器
  • 4.1.2 Name, Role, Value(A)— 自訂元件(custom radio、card 點擊區)有正確 accessible name + role

Form-payment 特別補充(testing-flow 已加入,本範本特別關注)

  • 1.3.5 Identify Input Purpose(AA)— 常見欄位(email、name、tel、address)加 autocomplete 屬性,讓密碼管理器/瀏覽器正確 autofill
  • 1.4.12 Text Spacing(AA)— 使用者覆寫文字間距(行高 1.5x、段距 2x、字距 0.12em、詞距 0.16em)時內容不丟失
  • 3.3.4 Error Prevention (Legal, Financial, Data)(AA)— 付款步驟可檢閱、可取消、可確認(三步)
  • 4.1.3 Status Messages(AA)— 「處理中」「成功」「失敗」用 role="status" 或 role="alert",讓螢幕閱讀器即時朗讀
字體大小 guardrail(非 WCAG 條文,實務 a11y 共識): body ≥ 16px(1rem)為絕對最低,建議 18px 對銀髮 audience 更友善。 Caption / small text ≥ 14px。 使用 rem / em 而非 px,讓使用者可在瀏覽器調大字體。
PART 1 · §C · 技術整合限制

技術整合限制 (高度可能,仍在確認)

狀態:高度可能,仍在確認中。若最終決策改變(例如改用其他平台或自建表單),Part 2 範例需要重新校準。設計階段以下述方向進行,實作前再次確認。
  • 報名表單:netiCRM iframe 嵌入 — no-code 路徑。netiCRM 後台建活動 → 產 embed URL → WordPress 端貼 iframe。視覺差異用 netiCRM 後台 CSS override 處理。
  • 付款金流:藍新(NewebPay)+ PayPal — 由 netiCRM 後台處理,網站端只負責導向 + 顯示結果。台灣付款走藍新,國際付款走 PayPal,贊助洽詢不走線上付款。
  • 會員資料、報名記錄、收據 — 統一進 netiCRM 後台,網站端不另建系統。
  • iframe a11y 要求 — iframe 必須加 title 屬性(例: 「ADALS 2026 報名表單」),內部表單必須通過 §B WCAG 清單(透過 netiCRM 後台 CSS override + 表單欄位設定)。
  • 切語言時 iframe 來源 URL 同步切換 — 中文表單與英文表單為不同 embed URL,語言切換時需替換。
PART 1 · §D · 內容能力要求

內容能力要求

雙語(zh / en)支援

  • 系統能力: 必須 — 切換語言功能必須存在(Polylang / WPML 或同等方案)。
  • 每個活動的雙語: 可選 — 不是所有活動都需要雙語。ADALS 2026 預計雙語;小型內部工作坊可能僅中文。活動管理者自行決定是否啟用該活動的英文版。
  • 設計考量:
    • 單語狀態: 介面不出現空英文 / 中文欄位(例如「英文標題:(未提供)」這種顯示)
    • 雙語狀態: 中英並列或可切換的排版仍需合理(英文標題通常更長,設計時要預留)
    • 網頁 <html lang> 屬性必須隨語言切換(讀屏會切換發音)

付款分流類型

  • 必須支援的分流類型: 台灣付款、國際付款、贊助洽詢、免費報名(無付款)
  • 每個活動啟用哪些: 由活動管理者決定。免費活動只啟用「免費報名」分流;ADALS 啟用所有四種。
  • 狀態頁: 四種必須各有對應: 處理中、驗證失敗、付款失敗、報名成功。可共用版面但訊息與行動方向必須清楚。
PART 1 · §E · 設計師自由空間

設計師自由空間(明列「不規定」的事)

以下事項本範本不規定,由設計師依視覺判斷與品牌規範決定:

  • 報名流程的步驟數量與排序(Part 2 示範 5 步驟,簡單活動可合併至 2-3 步)
  • 票種卡片的視覺樣式(欄位、邊框、選中狀態、價格顯示位置)
  • 表單欄位的排版(單欄 / 雙欄、欄位順序、grouping、必填標示樣式)
  • 狀態頁的視覺設計(僅要求訊息清楚 + WCAG 4.1.3)
  • 雙語切換按鈕的位置與樣式(頂部 / 側邊 / hamburger 內)
  • 按鈕樣式、間距、層級、hover / focus 細節(只要 focus 可見)
  • 字體選擇(已由品牌規範定義,範本不重複)
  • 動畫、轉場效果(只要尊重 prefers-reduced-motion)
  • 響應式斷點與佈局策略(只要 320px-1920px 範圍內可用、200% zoom 不爆版)
  • Part 2 提到的所有「示例」皆為一種可能,可全部重新設計

PART 2 · §3 STEP 1 · TICKET SELECTION

票種選擇示範 ADALS 範例,簡單活動可省略

DEMO · STEP 1 / 5
選擇票種
設計重點
  • <fieldset> + <legend> 包整組 radio，screen reader 會把整組讀為一個問題

PART 2 · §4 STEP 2 · REGISTRATION FORM

報名表單示範 參考範例

ADALS 報名表單應該盡量短，只問付款與通知絕對需要的資訊。其他偏好（飲食、住宿、議程關注）可以在報名成功後用後續 email 收集，不要塞進報名流程。

DEMO · STEP 2 / 5
將出現在報到名牌上。
電子郵件格式不正確，請輸入完整地址（例如 name@domain.com）。
用於議程後分組討論時的名牌資訊。
幫助主辦方安排議程互動分組。
無障礙需求（選填）
每個欄位都有理由
  • 姓名：報到名牌
  • Email：付款收據、議程提醒、無障礙需求確認
  • 服務單位：分組討論名牌（選填，避免無組織者卡關）
  • 角色身分：議程互動安排（5 選項即可，不要過細）
  • 無障礙需求：用 fieldset 包，避免被誤認為單選
a11y 重點
  • 每個 input 必須有 <label for>
  • 必填欄位用 aria-required="true"（不要只靠紅色 *）
  • 錯誤訊息用 aria-invalid + aria-describedby 連結到 role="alert"
  • autocomplete 屬性讓密碼管理工具與輔助技術可以填入正確內容（WCAG 1.3.5）
  • iOS Safari：input 必須 font-size: 16px 或以上，避免 focus 時自動放大頁面
不要犯的錯：用佔位符（placeholder）取代 label。Placeholder 在輸入後消失，老人和認知障礙使用者會忘記這個欄位是什麼。Label 必須一直在。
PART 2 · §5 STEP 4 · PAYMENT FLOW UI

付款流程的 UI 設計 ADALS 範例(三流),簡單活動可只用一流

付款不是一個畫面，是一個分流路由器。根據 Step 1 的票種選擇，導向三條不同的 UI 流程。本節聚焦 Christina 在每條流程要設計的 前後 wrapper UI(引導頁 + 確認頁)。

Christina 不負責的事:金流 API 串接、callback、簽章驗證、合規收據生成 — 這些由 netiCRM 內建處理或工程獨立介入。每條分流為什麼選某金流商、費率比較、銀行帳戶決策等,完整論述見 金流基礎建設決策依據。
票種 → UI 分流
一般票 / 早鳥 / 學生票 → 分流 A:跳轉到台灣金流付款頁,完成後跳回
國際票(USD) → 分流 B:跳出國際付款 checkout,雙語確認
企業贊助洽詢 → 分流 C:不上線收,跳轉到贊助洽詢表單(完全在 Christina scope 內)

分流 A:台灣門票

CHRISTINA 的 UX 設計

付款前引導頁

告知使用者「將前往金流商付款頁,完成後自動跳回 ADALS 網站」,提供「確認跳轉」按鈕。

付款後確認頁

顯示成功/失敗狀態、訂單編號、後續資訊(收據寄送、議程連結)。

付款頁本身由金流商代收(藍新 hosted),Christina 不能客製。設計範圍是跳轉前引導頁與跳轉後確認頁這兩個 wrapper。

a11y 注意(Christina 範圍內)
  • 跳轉前明確告知「將被導向 [金流商] 付款頁」(用文字,不只圖示)
  • 付款失敗回來時,前面填的資料要保留
  • 確認頁要有清楚的成功 / 失敗訊息,不要只顯示 status code
  • 跳轉按鈕要可鍵盤操作 (Tab + Enter)

分流 B:國際門票

CHRISTINA 的 UX 設計

國際票引導頁(英文為主)

明確標示美元計價(USD 100),告知將跳出至國際金流商 checkout。

付款後確認頁(雙語)

英文為主、中文為輔的成功訊息;提供報名編號、英文收據寄送說明。

金流嵌入由 netiCRM 內建或工程介入。Christina 不負責 SDK 嵌入與 webhook,只負責前後 wrapper UI。

a11y 注意(Christina 範圍內)
  • 確認頁雙語切換要同步更新 <html lang>
  • 英文文案不要直接 Google 翻譯,專業會議的英文需校對
  • 時區、幣別、稅務說明都需要清楚標示(海外與會者敏感)

分流 C:贊助洽詢(不走線上付款)

Elementor 預設只處理「正常」狀態。但實際使用時，使用者會遇到很多非正常狀態。每個狀態都需要明確的視覺與文字提示。

Scope 拆分：Christina vs. 平台 / 工程

Christina 的職責（Elementor + Ally 內，無需寫 code）

  • Hero / About / Programme / Audience / Sponsorship / Venue 等視覺頁面
  • 票種選擇 UI（若採自建路徑）或 netiCRM 嵌入位置（若採 netiCRM）
  • 報名表單欄位視覺 + 引導文字 + 必填標記 + 錯誤提示文案
  • 付款前引導頁、付款後確認頁的視覺設計
  • 錯誤頁、成功頁、loading 狀態的視覺呈現
  • WCAG 2.2 AA 視覺合規（對比度、focus ring、觸控目標 ≥ 44×44）
  • 中英雙語切換的視覺與互動

平台 / 工程的職責（Christina 不碰，超出 Elementor 範圍）

  • 藍新（NewebPay）/ PayPal / 其他金流的 API 串接
  • 付款 callback、簽章驗證、訂單 idempotency、refund 處理
  • 台灣稅法合規收據生成（統編、捐款扣抵編號、社團法人格式）
  • 會員資料庫 schema、跨年度資料管理
  • SWIFT 電匯流程與英文 invoice 生成（祕書長 / 總務線下手動）
  • 寄信系統（確認信、收據、提醒）

路徑依賴：「平台 / 工程的職責」具體由誰承擔取決於理事會決定的金流路徑： netiCRM 主推 → 平台內建處理； 自建備案 → Bertram 或工程資源獨立介入。 不論哪條路徑，Christina 的 scope 都不變。

表單服務選擇

建議

  • Fillout（fillout.com）— 有 WCAG 公開承諾，可嵌入 iframe，logic branching 支援良好
  • Cognito Forms（cognitoforms.com）— 有 a11y 聲明，付費版可串金流
  • 若 Christina 想用 Elementor 原生表單：用 Elementor Pro Forms + Ally 外掛 補強，不要用 Contact Form 7（a11y 不及格）

不要用

  • Typeform — 動畫過度、鍵盤導航有缺陷、screen reader 體驗差
  • KKTix / Accupass — 內部評估可及性不可接受
  • Eventbrite — 跨境費率累計 6–8%、版型限制論述呈現
  • Google Forms — 無法嵌入金流，且樣式無法符合 ADALS 設計風格

WCAG 2.2 AA 表單檢查清單

  • 所有 input 有 <label for>，不用 placeholder 取代 label（1.3.1, 3.3.2）
  • 必填欄位用 aria-required="true"（不只靠紅色 *）
  • 錯誤訊息用 role="alert" 或 aria-live="assertive"（3.3.1）
  • 錯誤訊息明確指出是什麼錯 + 怎麼修正（3.3.3）
  • autocomplete 屬性正確使用（1.3.5）
  • Radio / Checkbox 群組用 <fieldset> + <legend>
  • 所有互動元素 focus 可見（2.4.7），不要 outline: none
  • 觸控目標 ≥ 44×44 CSS pixels（2.5.8）
  • iOS 防自動縮放：input font-size: 16px 以上
  • 表單可用鍵盤完整操作（2.1.1）
  • 送出按鈕避免雙擊重複送出（disable on submit）
  • 顏色對比 ≥ 4.5:1（文字）、3:1（UI 元件邊框）

雙語切換

使用 data-zh="..." / data-en="..." 屬性，JS 切換時必須同步更新 <html lang> 屬性（讓 screen reader 用對的語言發音）。Form label、error message、button text 都要支援雙語切換。
